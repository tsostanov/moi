# Лабораторная работа №4

## Тема

Построение изображения трехмерной сцены методом трассировки путей с учетом глобального освещения.

## Что реализовано

Программа `main.py` строит изображение сцены из треугольной сетки методом path tracing. Сцена по умолчанию похожа на Cornell box: пол, потолок с проемом под световую панель, задняя стенка, цветные боковые стены, протяженный потолочный источник света и два объекта с разными материалами. Все поверхности заданы треугольниками.

Основные элементы реализации:

- цветовая модель RGB;
- пересечение луча с треугольником по алгоритму Меллера-Трумбора;
- BVH/AABB-дерево для ускорения поиска пересечений и теневых лучей;
- кеширование ребер, ограничивающего параллелепипеда и центра каждого треугольника;
- опциональный C-ускоритель для пересечения луча с треугольниками в маленьких сценах;
- диффузное отражение по Ламберту;
- идеальное зеркальное отражение;
- смешанный материал с диффузной и зеркальной составляющими;
- проверка физичности материала: `diffuse + specular <= 1` по каждому RGB-каналу;
- протяженный цветной источник света из двух треугольников;
- источник встроен в отверстие потолка, поэтому визуально выглядит как световая панель, а не как отдельный прямоугольник в воздухе;
- ламбертов источник: постоянная яркость по поверхности и множитель `cos(theta)` в геометрическом члене;
- выбор источника света по мощности;
- случайная точка на треугольнике источника;
- прямое освещение одним случайным лучом к источнику;
- непрямое освещение через трассировку путей;
- выбор события отражения по вероятностям, зависящим от диффузной и зеркальной составляющих;
- русская рулетка после заданной глубины;
- антиалиасинг случайной точкой внутри пикселя;
- нормировка абсолютных яркостей, отсечение выше 1, гамма-коррекция и запись PPM.

## Запуск

Минимальный запуск:

```bash
python 04/main.py
```

Запуск с оконным интерфейсом:

```bash
python 04/gui.py
```

В `gui.py` можно менять разрешение, число сэмплов, глубину трассировки, русскую рулетку, число worker-процессов, размер чанка строк, сцену, материал, камеру, свет, OBJ-файл и путь вывода. Кнопка `Start render` запускает `main.py` в отдельном процессе, лог рендера показывается в окне. В Windows кнопка `Stop` останавливает текущий рендер вместе с дочерними worker-процессами. Результат сохраняется в PPM, PNG и TXT так же, как при запуске из консоли.

В интерфейсе есть пресеты:

- `Preview` - быстрый предпросмотр 240x240 с малым числом сэмплов;
- `Balanced` - умеренный рендер 500x500 для проверки результата;
- `Final` - более тяжелый рендер 500x500 для сдачи.

По умолчанию используется `500x500`, `1 spp`, `max-depth 2`, чтобы расчет не был слишком долгим. Для более чистой картинки увеличивайте `--samples`.

При запуске без параметров результат сохраняется в:

```text
04/outputs/render.ppm
04/outputs/render.png
04/outputs/render.txt
```

Файл `render.ppm` содержит изображение в формате PPM P6. Файл `render.png` создается автоматически из тех же нормированных и gamma-corrected RGB-данных, поэтому его удобно открывать обычным просмотрщиком и вставлять в отчет. Файл `render.txt` содержит параметры запуска, число треугольников, число источников, нормировочный коэффициент и список материалов.

В репозитории для сдачи оставлены два готовых набора:

```text
04/outputs/final.ppm
04/outputs/final.png
04/outputs/final.txt
04/outputs/test.ppm
04/outputs/test.png
04/outputs/test.txt
```

`final.png` - основная демонстрационная картинка для отчета. `final.ppm` - тот же результат в формате, требуемом заданием. `final.txt` содержит параметры финального запуска.

## Примеры для защиты

Важно: в `--output` лучше указывать путь к `.ppm`. PNG-файл создается автоматически рядом с тем же именем.

Быстрый тест с маленьким разрешением:

```bash
python 04/main.py --width 240 --height 240 --samples 2 --max-depth 3 --rr-depth 3 --workers 4 --chunk-rows 24 --output 04/outputs/test.ppm
```

Финальный рендер с допустимым разрешением:

```bash
python 04/main.py --width 500 --height 500 --samples 32 --max-depth 6 --rr-depth 3 --white-point 3.2 --workers 4 --chunk-rows 16 --output 04/outputs/final.ppm
```

Более быстрый финальный вариант, если время расчета ограничено:

```bash
python 04/main.py --width 500 --height 500 --samples 16 --max-depth 4 --rr-depth 3 --workers 4 --chunk-rows 16 --output 04/outputs/final.ppm
```

Если нужно ускорить расчет на многоядерном процессоре:

```bash
python 04/main.py --width 500 --height 500 --samples 32 --max-depth 6 --rr-depth 3 --white-point 3.2 --workers 4 --chunk-rows 16 --output 04/outputs/final.ppm
```

## Опциональный C-ускоритель

Основная версия работает без компиляции, только на Python. Дополнительно есть небольшой C-модуль `native_intersect.c`: он реализует горячую часть Меллера-Трумбора для пачки треугольников и подключается через `ctypes`, если рядом с `main.py` лежит подходящая библиотека `native_intersect.dll` или `native_intersect.so`.

Сборка:

```bash
python 04/build_native.py
```

На Windows Python и компилятор должны быть одной разрядности. Для 64-bit Python нужен 64-bit компилятор, например MSVC Build Tools или MinGW-w64 x86_64. Если библиотека не собрана или не загрузилась, программа автоматически использует Python/BVH-реализацию.

Параметр `--chunk-rows` управляет размером задания для worker-процесса. Слишком маленькое значение чаще печатает прогресс, но увеличивает накладные расходы multiprocessing. Для 500x500 обычно удобно `16` или `32`:

```bash
python 04/main.py --width 500 --height 500 --samples 32 --max-depth 5 --rr-depth 3 --workers 4 --chunk-rows 16 --output 04/outputs/final.ppm
```

Сравнение шума при разном числе лучей на пиксель:

```bash
python 04/main.py --width 500 --height 500 --samples 4 --workers 4 --output 04/outputs/render_4spp.ppm
python 04/main.py --width 500 --height 500 --samples 32 --workers 4 --output 04/outputs/render_32spp.ppm
```

Сравнение материалов:

```bash
python 04/main.py --material-mode diffuse --output 04/outputs/diffuse.ppm
python 04/main.py --material-mode mirror --output 04/outputs/mirror.ppm
```

Смена сцены, камеры и света:

```bash
python 04/main.py --scene mirror-test --camera 0 1.2 3.6 --look-at 0 0.9 0 --light-scale 1.4 --output 04/outputs/mirror_test.ppm
```

Дополнительно можно вставить OBJ-модель:

```bash
python 04/main.py --obj model.obj --obj-scale 0.4 --obj-offset 0 0 0.2
```

## Алгоритм

Для каждого пикселя генерируется несколько первичных лучей. Начало луча находится в точке камеры, а направление проходит через случайную точку внутри пикселя. Это дает антиалиасинг.

Далее для луча ищется ближайшее пересечение с треугольниками сцены. Для ускорения строится BVH/AABB-дерево: сначала луч проверяется против ограничивающих параллелепипедов узлов, и только потом против треугольников в подходящих листьях. Если луч попал в протяженный источник света, его яркость добавляется к результату для первичного или зеркального луча. Если луч попал в обычную поверхность, программа отдельно оценивает прямой свет: выбирает один источник с вероятностью, пропорциональной его мощности, выбирает случайную точку на этом источнике и проверяет видимость теневым лучом через то же BVH-дерево.

После прямого освещения выбирается следующее событие пути:

- диффузное отражение выбирается с вероятностью, зависящей от яркости диффузной RGB-составляющей;
- зеркальное отражение выбирается с вероятностью, зависящей от яркости зеркальной RGB-составляющей.

Для диффузного отражения используется косинусная выборка полусферы, поэтому множитель `cos(theta)` сокращается с плотностью выборки. Для зеркального отражения направление вычисляется как идеальное отражение относительно нормали.

После глубины `--rr-depth` применяется русская рулетка. Если путь выживает, его вклад делится на вероятность выживания, поэтому оценка остается несмещенной.

## Формирование изображения

Сначала накопленные значения хранятся как линейные абсолютные яркости RGB. Затем выполняется:

1. поиск нормировочного значения;
2. перевод в относительные яркости делением на нормировочное значение;
3. отсечение значений выше 1;
4. гамма-коррекция `1 / 2.2`;
5. перевод в диапазон 0-255;
6. запись файла PPM;
7. запись PNG-копии для просмотра.

Если нужно зафиксировать одинаковую нормировку для нескольких картинок, можно передать `--white-point`.

## Что говорить на защите

| Требование | Где в коде | Короткий ответ на защите |
|---|---|---|
| Сцена из треугольной сетки | `Triangle`, `add_quad`, `add_box`, `add_pyramid`, `load_obj_triangles` | Вся геометрия хранится как список треугольников. Даже стены и объекты разбиты на треугольники, также есть загрузка OBJ. |
| Пересечение луча с поверхностью | `intersect_triangle`, `Scene._intersect_bvh`, `native_intersect.c` | Используется алгоритм Меллера-Трумбора. Для ускорения сцена предварительно разбивается BVH/AABB-деревом, а маленькие сцены могут проверяться C-функцией через `ctypes`. |
| RGB-расчет | `Vec3` | Цвета материалов, источников и накопленная яркость представлены тремя компонентами RGB в линейном пространстве. |
| Диффузный материал | `Material.diffuse`, `estimate_direct_light`, `sample_cosine_hemisphere` | Диффузная BRDF равна `diffuse / pi`, направления непрямого луча выбираются по косинусному распределению. |
| Зеркальный материал | `Material.specular`, `reflect` | Для зеркального события направление отраженного луча считается как идеальное отражение относительно нормали. |
| Смешанный материал | `Material(diffuse, specular)` и `--material-mode balanced` | Поверхность может одновременно иметь диффузную и зеркальную RGB-составляющие. Тип следующего события выбирается случайно по вероятностям. |
| Физическая корректность | `Material.__post_init__` | При создании материала проверяется, что по каждому каналу `diffuse + specular <= 1`, поэтому поверхность не отражает больше энергии, чем получила. |
| Выбор события отражения | `trace_path`, `specular_probability` | Вероятность зеркального события берется из яркости зеркальной составляющей, иначе выбирается диффузное событие. Вклад делится на вероятность выбора. |
| Русская рулетка | `trace_path`, блок `if depth + 1 >= rr_depth` | После заданной глубины путь может завершиться случайно. Если путь выжил, throughput делится на вероятность выживания, поэтому оценка остается несмещенной. |
| Точечная камера | `Camera.look_at`, `Camera.make_ray` | Камера задается одной точкой, направлением на цель и углом обзора. Первичный луч выходит из точки камеры. |
| Антиалиасинг | `Camera.make_ray` | Для каждого сэмпла используется случайное смещение `jitter_x`, `jitter_y` внутри пикселя. |
| Протяженный источник света | материал `area light`, потолочный quad из двух треугольников | Источник имеет площадь, цветную яркость и выбираемую случайную точку на поверхности, поэтому тени получаются мягкими. |
| Выбор источника по мощности | `Scene.rebuild_lights`, `Scene.sample_light` | Для светящихся треугольников строится CDF с весом, пропорциональным площади и яркости источника. |
| Прямое освещение | `estimate_direct_light` | В каждой точке выбирается один источник, случайная точка на нем и проверяется видимость теневым лучом. |
| Глобальное освещение | `trace_path` | После попадания в поверхность луч продолжает путь через диффузное или зеркальное отражение, поэтому учитываются переотражения света. |
| Критерий остановки | `--samples`, `--max-depth`, `--rr-depth` | Качество и время расчета управляются числом лучей на пиксель, максимальной глубиной и глубиной включения русской рулетки. |
| Формирование изображения | `write_images`, `build_display_bytes`, `write_ppm`, `write_png` | Сначала считается линейная яркость, затем выполняются нормировка, отсечение выше 1, гамма-коррекция и запись PPM. PNG создается дополнительно для просмотра. |
| Интерфейс | `parse_args` | Интерфейс командной строки позволяет менять камеру, свет, сцену, материал, размер, число сэмплов, глубину и выходной файл. |

## Вывод

Реализован базовый физически осмысленный path tracer для треугольной сцены. При увеличении `--samples` шум уменьшается, протяженный источник дает мягкие тени, цветные стены дают цветные переотражения, а зеркальная составляющая меняет вид объектов через отраженные лучи.
